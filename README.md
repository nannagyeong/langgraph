# LangGraph 기반 멀티 에이전트 운동 추천 시스템

사용자의 운동 관련 질문을 입력받으면, 질문 속 상태를 먼저 파악하고 그에 맞는 운동 후보를 추천한 뒤 최종적으로 정리된 답변을 생성하는 프로젝트입니다.  
LangGraph를 사용해서 **질문 분석 → 운동 추천 → 답변 생성** 흐름을 단계별로 분리했고, Ollama의 로컬 LLM을 연결해 멀티 에이전트 구조를 직접 구현했습니다.

---

## 프로젝트 소개

처음에는 운동 추천 질문을 하나의 프롬프트로 바로 처리하는 방식으로 구현했습니다.  
그런데 그렇게 해보니 질문 안에 들어 있는 상태 정보와 추천 결과가 한 번에 섞이면서, 응답 형식이 일정하지 않거나 추천 운동 강도가 사용자 상황과 잘 맞지 않는 경우가 있었습니다.

예를 들어 아래 같은 질문이 들어왔을 때,

> 체력이 안좋고, 살이 계속 찌는데 어떤 운동을 할까?

모델이 질문 전체를 한 번에 처리하면서 현재 상태를 정확히 정리하기보다, 바로 운동을 나열하는 식으로 응답하는 경우가 많았습니다.  
그래서 질문을 한 번에 처리하는 대신, 아래처럼 역할을 나눠서 처리하는 구조로 바꿨습니다.

- 질문에서 현재 상태 추출
- 상태에 맞는 운동 후보 추천
- 추천 결과를 바탕으로 최종 응답 생성

이렇게 나누고 나니 각 단계의 역할이 더 분명해졌고, 결과가 이상하게 나왔을 때 어느 부분을 수정해야 하는지도 훨씬 명확해졌습니다.

---

## 왜 이 프로젝트를 만들었는가

이 프로젝트는 단순히 운동 추천 결과를 만드는 것보다,  
**LLM 기반 작업을 단계별로 분리했을 때 얼마나 더 통제 가능하게 만들 수 있는지** 확인해보기 위해 시작했습니다.

특히 아래 두 가지를 직접 확인해보고 싶었습니다.

1. 하나의 LLM에게 모든 역할을 맡기는 방식보다, 역할을 분리했을 때 응답 품질이 더 안정적일까  
2. LangGraph의 상태 기반 워크플로우가 실제로 멀티 에이전트 구조를 설계하는 데 얼마나 유용할까

## 설계 방향

이번 프로젝트에서 가장 중요하게 본 건 **“한 번에 답하지 말고, 역할을 나눠서 처리하자”**는 점이었습니다.

운동 추천처럼 먼저 사용자의 상태를 읽어야 하는 질문은, 바로 답을 생성하게 하면 응답이 편하긴 하지만 다음 문제가 있었습니다.

- 상태 추출과 운동 추천이 섞여서 결과가 흐려짐
- 추천 기준이 일정하지 않음
- 출력 형식을 통제하기 어려움

그래서 워크플로우를 아래 3단계로 나눴습니다.

1. Extractor Agent: 질문에서 현재 상태 추출
2. Matcher Agent: 상태에 맞는 운동 후보 추천
3. Answer Agent: 최종 응답 생성
